Why the Dwell Mod Keeps Breaking — and How to Turn It Around The whack-a-mole isn't bad luck. It's one root cause repeating. Here's the model.

1 · The one idea that changes everything

WHAT WE ASSUMED

"The game can tell the mod when a screen opens or the state changes."

So we wrote code that waits to be notified, then scans. When the notice was missed, the overlay never appeared.

WHAT'S ACTUALLY TRUE

The game emits nothing to the mod. The mod is an outside observer.

The only way to know what's on screen is to look — walk the game's UI tree. There is no free clock, no notification. Timing must be discovered, and discovery costs a scan.

2 · What a "scan" really costs

A tree walk is not "look up a thing." It's visit every UI node in the game, one at a time, across the slow C#↔engine boundary.

3 · Every bug is the same shape

These two lines look identical to a reader:

FindAll<NPotion>(root) ← in a cached path = free

FindAll<NPotion>(root) ← runs every frame = disaster

The danger isn't in the line — it's in how often the function around it runs, which lives in a different file. That's why the cost is invisible and keeps getting underestimated.

The seesaw (your whack-a-mole):

You say "overlays missing" → safe fix = scan more → FPS drops.
You say "FPS dropped" → safe fix = scan less → overlays go missing.

Each fix is correct for the complaint in front of it and quietly breaks the other side. Nobody is ever looking at both at once.

4 · The fix: one coordinator, computed once per frame

Stop letting ~10 overlays each poke the tree on their own. Derive the state ONCE, in ONE place, and have everyone read the shared result.

Split the two jobs you correctly identified:

WHEN to scan = owned centrally (timing).  •  HOW to scan = owned by each overlay, but only runs when the coordinator says so.

① Coordinator

Once per frame: one cheap, visibility-pruned look. Produces a snapshot: mode, active screen id, did the screen change?, is a menu open?

→

② "Should I scan?"

Just compares numbers: did the screen's id change, or has ~1s passed? No tree walk to decide this.

→

③ Overlays react

If told yes → do ONE scoped scan & cache the rectangles. Every other frame → just draw the cached rectangles.

The single number that means "healthy":

walks-per-frame ≈ 0  on a screen that isn't changing. Detection is one cheap pass; everything else is reading cached rectangles.

5 · We do NOT start from scratch

KEEP — hard-won & valuable
✓Game knowledge: which node types mean what (upgrade screen, selected-card container, etc.)
✓Card measuring: rotated fan-layout math, holder anchoring
✓The confirm-vs-hover distinction we just untangled
✓The timing primitive (ScreenEntryScanState) & mode service — already ~80% of the coordinator
STRIP — the source of fragility
✕Every overlay detecting game state by itself
✕Hand-rolled cache flags (these caused the PauseMenu & DeckView per-frame storms)
✕3+ overlapping copies of "is this the confirm phase?"
✕Any full-tree walk reachable from a per-frame path

A full rewrite would throw away the left column and re-hit every bug in it. Instead we consolidate — make the coordinator the single source of truth, then move overlays onto it one at a time.

6 · The plan — and your play data is the ruler

  1. 1
    Instrument (your idea). Tag every tree walk with its caller + a per-frame budget alarm. Then real play data reads: walk[PauseMenu] calls/frame = 1.0 — the storm names itself, instead of us guessing for hours.
  2. 2
    Build the coordinator. One per-frame snapshot (mode, active screen, changed?, menu open?) from a single cheap pass.
  3. 3
    Migrate overlays one at a time. Each moves onto the coordinator; delete its private detection. After each, confirm from your play log that walks-per-frame stayed ≈ 0. Measured, not estimated.
Why this ends the cycle: it replaces estimation (which AI is bad at — it can't see caller frequency) with measurement (which is exact). You'll literally watch the cost drop, overlay by overlay, in data from your own play.
