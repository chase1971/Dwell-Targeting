Key idea: Frame cost ≈ how many NodeQuery.FindAll walks hit the scene root per frame. Each node visit crosses C#↔Godot (GetChildren). Static screens should scan once on entry + once when something changes (choice made, chest opened), not on a timer every 250–1000 ms.
Jul 2026 testing — what Scan Load can miss:
  • Duplicate call sites — same ROOT walk invoked 3–5× per frame from different guards (MainMenuOverlay.IsActive() is the worst offender). Multiply single-overlay load by call count.
  • Pre-gate walks — a 250 ms throttle on rebuild does nothing if detection (FindMainMenu, ComputeLayoutKey) runs every frame anyway. RoomOverlay had this bug; MainMenuOverlay still does.
  • Timer invalidation chains — GetMode() every 250 ms used to invalidate ViewScreenQuery, forcing a second ROOT walk. Fixed for stable screens via IsCachedModeStillValid().
  • Non-scan FPS cost — DwellDebugOverlay (green hitbox render) costs frames but scores 0 Scan Load. Toggle off to isolate tree-walk cost.
  • Pause fast path — Escape menu skips GetMode + full CollectDwellTargets; observed ~60 FPS even when title menu (~150k load) runs ~30–40.

Scan Load score — chip-away metric (5 seconds)

Formula: Load = scans_in_5s × tree_walks_per_fire × scope_points × call_mult

scans_in_5s: at 250 ms interval → 5000÷250 = 20 scans. At 500 ms → 10. Every frame (60 FPS) → ~300. stable (screen unchanged) → 0 additional scans.
scope_points: ROOT full-tree = 100, SCOPED subtree = 5, none = 0.
call_mult: how many times the same walk fires per frame from different call sites (default 1). Title menu FindMainMenu ≈ 4×.
Example: ROOT walk every 250 ms, 1 walk → 20 × 1 × 100 = 2,000. Scan once on room entry → 1 × 3 × 100 = 300, then 0 until chest clicked.
Observed Jul 2026: Treasure room steady ~60 FPS (~3.5k load). Card-grid upgrade was ~30 FPS at ~105k+ real load (CardConfirm ROOT ×2/frame + ViewScroll WalkForCardGrid) despite chart showing ~35k. Title menu ~30–40 FPS at ~150k+ (MainMenu pre-gate). Pause menu ~60 FPS (~2k).

This is not milliseconds or FPS — it's a relative score so you can say "treasure room was 105k, now ~3.5k after fixes." Lower is better. Sum only overlays active in each scenario.

↓

↓
UpdateFrame — SettingsOverlay, OverlayVisToggle, ModConfigScrollHarmonizer
↓

↓
Pause open — short path: PauseMenu + Back + gear only (~60 FPS observed)
Settings blocking — SuppressForMenu + empty dwell (no Finalize scans)
↓
Screen mode .Sync() — Rewards / PileSelect / Map / Event / Shop / Room
Combat path — HandSelect / CombatRows / EndTurn / Confirm
↓
FinalizeDwellTargets → DeckView + ViewScroll (cached) → CollectDwellTargets → ProcessFrame
↓
CardConfirmPhaseQuery — cached per screen (scoped walk once). Mode early-exit on Room/Map/Event/Shop. RED remaining: MainMenuOverlay pre-gate FindMainMenu.
